# 优化共享知识库

## 目标

本轮优化面向本地单用户场景，优先级如下：

1. 提升首次安装与首次初始化的易用性。
2. 降低重复操作、页面切换、长耗时任务导致的严重问题。
3. 让失败可理解、可恢复、可重试。
4. 不为当前场景过度设计云化或高并发能力。

## 当前确认的问题

### 1. 全量任务可重复启动

- `POST /api/v1/tasks/start` 当前没有运行中任务拦截。
- `TaskService.create_task()` 当前没有本机互斥保护。
- 在本地多开页面或重复点击时，可能并发跑多套 `run_all.py`。

### 2. 请求没有真正取消

- 前端 `axios` 实例没有统一接入 `AbortController`/`signal`。
- 当前主要依赖 `loading` 标志位、请求序号、忽略旧响应来缓解竞态。
- 这样可以减少脏写，但不能减少后端资源浪费。

### 3. 初始化完成态定义不统一

- `configStore.dataInitialized` 依赖 `raw_data + candidates + analysis`。
- `Update.vue` 的 `bootstrapFinished` 只依赖 `rawData.exists`。
- 用户可能看到“初始化已完成”，但别的页面仍提示“请先完成首次初始化”。

### 4. 长耗时流程反馈仍不够完整

- 当前已有任务中心、任务日志、任务阶段。
- 但首次初始化仍缺少完整的预期说明、阶段解释、失败恢复入口、刷新恢复能力。

## 非目标

- 不做分布式锁。
- 不做云服务化改造。
- 不以大规模并发为目标重构数据库层。
- 不引入重量级任务队列。

## 本轮优化范围

### A. 后端任务互斥

- 阻止全量初始化/全量更新重复启动。
- 返回明确的“已有任务运行中”错误或已有任务信息。
- 尽量保证刷新页面后仍可恢复查看当前任务。

### B. 初始化状态统一

- 统一“是否完成首次初始化”的判定来源。
- App 顶部状态、任务中心引导、功能页拦截使用相同定义。

### C. 请求取消

- 页面切换、重复触发、条件切换时可取消旧请求。
- 至少覆盖 `TomorrowStar`、`Diagnosis`、`Watchlist` 的长请求路径。

### D. 首次初始化体验

- 明确展示当前步骤、预计影响、失败后的恢复动作。
- 支持页面刷新后恢复任务视图。
- 保持本地服务风格，避免复杂运维术语。

## 约束

- 不回滚现有用户改动。
- 优先做增量改造，保持现有接口和结构基本稳定。
- 前后端改动应尽量可回归测试。

## 验收标准

1. 不能重复启动全量初始化/全量更新。
2. “初始化是否完成”在任务中心、App 顶部、业务页表现一致。
3. 页面切换或重复操作不会继续保留明显无效的旧请求。
4. 首次初始化流程能回答四个问题：
   - 现在卡在哪一步？
   - 为什么还不能用？
   - 失败后去哪里恢复？
   - 刷新页面后能不能继续看？

## 完成情况

- 已完成本机单任务互斥：重复点击全量初始化/全量更新会复用当前任务，不再重复启动。
- 已统一首次初始化口径：以“原始数据、候选结果、分析结果均就绪”为完成条件。
- 已完成页面作用域请求取消：覆盖 `TomorrowStar`、`Diagnosis`、`Watchlist` 关键长请求路径。
- 已完成首次初始化引导与刷新恢复：配置页和任务中心都补齐了首次使用路径、失败恢复和日志恢复。

## 当前残留风险

- 互斥仍是本地单进程方案，不覆盖多进程/多实例。
- 前端取消请求只能取消浏览器端等待，后端已启动的任务未必停止。
- 初始化任务视图恢复依赖 `sessionStorage`，不跨标签页共享。
- 现有前端 Vitest 用例与当前实现有明显漂移，需要单独整理。
